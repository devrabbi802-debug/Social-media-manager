DATABASE SCHEMA (All Tenant DB Tables)
---------------------------------------

categories:
  - id, parent_id (self-FK nullable), name, slug, description, image
  - sort_order, is_active, timestamps
  - Unique: (slug, parent_id)

attribute_templates:
  - id, category_id (FK nullable), is_global (bool default false)
  - name, slug, type (enum: text/number/select/boolean/date)
  - options (JSON nullable), is_required, sort_order
  - Unique: (category_id, slug, is_global)

brands:
  - id, name (unique), slug (unique), logo, is_active

products:
  - id, category_id (FK), brand_id (FK nullable)
  - name, slug (unique), sku (unique), description
  - base_price (decimal 12,2), discount_price (decimal 12,2 nullable)
  - stock_quantity (int default 0), unit (default 'pcs'), barcode
  - status (enum: active/inactive/out_of_stock)
  - is_featured, meta_title, meta_description, sort_order

product_attribute_values:
  - id, product_id (FK), attribute_template_id (FK), value (text)
  - Unique: (product_id, attribute_template_id)

product_variants:
  - id, product_id (FK), sku (unique), name (nullable)
  - price (decimal nullable), stock_quantity, attributes (JSON)
  - barcode, is_active

product_images:
  - id, product_id (FK), image_path, alt_text, sort_order
  - image_analysis (JSON nullable — Gemini AI output)

warehouses:
  - id, name, address, phone, is_active

stock_movements:
  - id, product_id (FK), variant_id (FK nullable), warehouse_id (FK)
  - type (enum: in/out/adjustment), quantity, reference, notes
  - created_by (FK nullable)

inventory_alerts:
  - id, product_id (FK unique), threshold (default 10), is_active

2. CRITICAL BUGS FOUND
================================================================================

BUG #1: STOCK IN/OUT VARIANT RECALCULATION MISSING
---------------------------------------------------
Location: InventoryController::stockIn(), InventoryController::stockOut()
Problem:  When variant_id is provided, only the variant stock_quantity is
          updated. Product's stock_quantity (which should = sum of variant stocks)
          is NOT recalculated.
Impact:   Product-level stock shows wrong number when variant stock changes.
Fix:      Call recalculateProductStock() after variant stock changes.

BUG #2: NO TRANSACTION WRAPPING ON STOCK OPERATIONS
----------------------------------------------------
Location: ProductController (stockIn, stockOut, adjustStock)
Problem:  Stock increment/decrement + StockMovement creation are separate DB
          operations. If StockMovement fails, stock is already changed with no
          audit trail.
Impact:   Stock counts can drift from actual records.
Fix:      Wrap in DB::transaction().

BUG #3: EDIT PAGE ATTRIBUTE STALE VALUE ISSUE
----------------------------------------------
Location: ProductController::update() attribute handling
Problem:  When editing a product, only non-empty attribute values are saved.
          If user clears an attribute value, the old value REMAINS in the DB
          because empty values are skipped.
Impact:   Products show outdated attribute information.
Fix:      Delete existing attribute values before re-saving new ones.

BUG #4: QUICK STOCK FORMS DON'T SUPPORT VARIANTS
-------------------------------------------------
Location: Inventory index blade view
Problem:  Quick stock in/out/adjust forms only let you select a product, not
          a specific variant. If product has variants, adjustment goes to parent.
Impact:   Variant-level stock management is impossible from dashboard.
Fix:      Add variant selector in quick forms when product has variants.

BUG #5: CREATE PAGE ONLY SHOWS ROOT CATEGORIES
----------------------------------------------
Location: ProductController::create()
Problem:  Category dropdown uses Category::whereNull('parent_id') — only shows
          root categories. Subcategories cannot be assigned during product creation.
Impact:   Products can only be in top-level categories, not subcategories.
Fix:      Show all active categories with hierarchy (indented dropdown).

BUG #6: LOW STOCK ALERT IGNORES VARIANTS
-----------------------------------------
Location: InventoryAlert::isLowStock()
Problem:  Only checks product.stock_quantity. If product has variants, the
          relevant stock is on individual variants.
Impact:   Low stock alerts don't trigger for variant-level stockouts.
Fix:      Check variant stock if product has variants.

BUG #7: ADJUSTSTOCK ALLOWS ZERO QUANTITY
-----------------------------------------
Location: InventoryController::adjustStock() validation
Problem:  Validation allows quantity 0, creating meaningless adjustment records.
Impact:   Noise in stock movement history.
Fix:      Add min:1 validation.

BUG #8: VARIANT EDIT MODAL URL BUG
-----------------------------------
Location: products/show.blade.php line ~299
Problem:  variantForm.action URL has string concatenation error — missing closing
          quote on $product->id. Also uses hardcoded /dashboard/products/
          instead of route helper.
Impact:   Variant edit modal may not work correctly.
Fix:      Fix URL construction and use route() helper.

BUG #9: BRAND SLUG NOT VALIDATED
---------------------------------
Location: BrandController::store()
Problem:  Validates name unique but not slug. DB has unique constraint on slug.
          Two brands with same name-derived slug could pass validation but fail DB.
Impact:   Unexpected database errors.
Fix:      Add slug unique validation.

BUG #10: PRODUCT BOOT HOOK CONFLICT
------------------------------------
Location: Product model boot() + Controller store/update
Problem:  Model's booted hook updates status BEFORE controller validation runs.
          Status could be forced to 'out_of_stock' even if user selected 'active'.
Impact:   User's explicit status choice overridden silently.
Fix:      Move status logic to controller after validation.

3. HOW PROFESSIONAL PLATFORMS WORK
================================================================================

A. SHOPIFY — Product Variant System (Industry Leader)
-----------------------------------------------------

Product Data Model:
  Product
    ├── ProductVariant (up to 2,048 per product)
    │     ├── SKU, price, inventory_item
    │     └── selectedOptions: [Color: Red, Size: M]
    └── Options: max 3 (e.g., Color, Size, Material)

Key Features:
  - Product Taxonomy: 10,000+ categories, 1,000+ attributes
  - Auto-category → auto-attribute assignment
  - Vision Language Models: AI analyzes product image → suggests category + attributes
  - 85% acceptance rate of AI-suggested categories
  - 30M+ predictions processed daily
  - Variant matrix: auto-generated from option combinations
  - Each variant: own SKU, price, stock, image, barcode

Product Creation Flow (Shopify Standard):
  Step 1: Title + Description
  Step 2: Media (Images/Video) — AI auto-analyze
  Step 3: Pricing (Base + Compare-at price)
  Step 4: Inventory (SKU, barcode, quantity)
  Step 5: Shipping (weight, dimensions)
  Step 6: Options (Color, Size, Material) → Auto-generate variants
  Step 7: SEO (meta title, description)
  Step 8: Status (Active/Draft/Archived)

How Variants Work:
  1. User defines "Options" (e.g., Color: Red,Blue | Size: S,M,L)
  2. System auto-generates all combinations (2 colors × 3 sizes = 6 variants)
  3. Each variant gets: SKU (auto or custom), price, stock, image
  4. Variant matrix view: grid showing all combinations
  5. Can edit individual variant details inline

AI Inventory Features (2026):
  - Demand forecasting (sales + trends + weather + promotions)
  - Floating reorder points (adapt in real-time)
  - Exception monitoring ("SKU-A will stockout in 11 days")
  - Auto PO generation when stockout predicted
  - Multi-channel inventory sync
  - 15-25% reduction in inventory costs reported

B. WOOCOMMERCE — Variable Product System
-----------------------------------------

Product Types:
  - Simple Product
  - Variable Product (has variations)
  - Grouped Product
  - External/Affiliate Product

How Variants Work:
  1. Create "Variable Product"
  2. Add "Attributes" (e.g., Color: Red | Blue, Size: S | M | L)
  3. Check "Used for variations" on each attribute
  4. Click "Create variations from all attributes"
  5. Auto-generates: Red-S, Red-M, Red-L, Blue-S, Blue-M, Blue-L
  6. Each variation: SKU, price, stock, weight, dimensions, image

Database Architecture:
  - Variations stored as separate WP posts (post_type: product_variation)
  - Attributes stored as custom taxonomies
  - Product Attribute Values in wp_postmeta
  - Parent-child relationship via post_parent

Performance Notes:
  - Degrades beyond 50 active variations without caching
  - Large stores need denormalized product index table
  - Elasticsearch recommended for 100K+ products

C. MAGENTO — Configurable Product System (EAV Model)
-----------------------------------------------------

Product Types:
  - Simple Product
  - Configurable Product (has options/variants)
  - Bundle Product
  - Grouped Product
  - Virtual Product
  - Downloadable Product

EAV Model (Entity-Attribute-Value):
  - catalog_product_entity: main product table
  - catalog_product_entity_text: text attributes
  - catalog_product_entity_decimal: decimal attributes
  - catalog_product_entity_int: integer attributes
  - catalog_product_entity_varchar: varchar attributes
  - eav_attribute: attribute definitions
  - eav_attribute_option: attribute option values
  - eav_attribute_option_value: option value translations

Attribute Sets:
  - Group related attributes (e.g., "T-Shirt" set: Color, Size, Material)
  - Product assigned to attribute set during creation
  - Only relevant attributes shown based on attribute set
  - Unlimited attributes per product (vs Shopify's 3 options)

Configurable Products:
  - Parent product = configurable
  - Child products = simple products (each variant is independent)
  - catalog_product_super_attribute: links parent to attribute
  - catalog_product_super_link: links parent to child products
  - Each child: own SKU, price, stock, SEO

Performance:
  - Flat catalog indexing required for fast storefront
  - Reduces variant page load from 800ms to <200ms
  - Supports unlimited variants (no hard limit)

D. DARAZ — Marketplace Product Management
------------------------------------------

Product Creation:
  - Category selection → auto-loaded attributes
  - Brand selection (from predefined list)
  - Product title + description
  - Variant creation: Color, Size, etc.
  - Each variant: SKU, price, stock, images
  - Seller manages inventory per variant

Inventory Features:
  - Real-time stock sync across channels
  - Low stock alerts
  - Bulk inventory upload via CSV
  - Warehouse-wise inventory management
  - Auto-deduct on order confirmation

YOUR TOP 5 GAPS:

1. VARIANT MATRIX GENERATION (Critical)
   - Professional: Auto-generate all combinations from options
   - You: User manually creates each variant, adds attributes one by one
   - Impact: User confusion, error-prone, slow product creation

2. UNIFIED ATTRIBUTE SYSTEM (Critical)
   - Professional: Same attribute system for products AND variants
   - You: Two separate systems (ProductAttributeValue + Variant JSON)
   - Impact: Data inconsistency, no filtering on variant attributes

3. CATEGORY → AUTO ATTRIBUTES (Medium)
   - Professional: Select category → relevant attributes auto-load
   - You: User manually creates AttributeTemplate, then assigns
   - Impact: Extra work, possible missing attributes

4. STOCK OPERATIONS SAFETY (Critical)
   - Professional: DB transactions, atomic operations
   - You: No transactions, race conditions possible
   - Impact: Stock count corruption

5. AI DEMAND FORECASTING (Future)
   - Professional: AI predicts stockout, auto-generates PO
   - You: Basic low-stock threshold only
   - Impact: Reactive vs proactive inventory management
