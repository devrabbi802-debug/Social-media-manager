### DECISION 1: Extra Fields Storage Model
-------------------------------------------

FINAL RULE:

  BusinessCategory.extra_fields JSON = FIELD DEFINITION (schema)
  attribute_templates table = TEMPLATE STORAGE (created ONCE per category)
  product_attribute_values table = DATA STORAGE (per product)

FLOW:
  1. BusinessCategory.extra_fields e define thakbe:
     [
       {"name": "size_chart", "label": "Size Chart", "type": "textarea", "required": true},
       {"name": "fabric_type", "label": "Fabric Type", "type": "select", "options": ["Cotton","Polyester"]}
     ]

  2. AttributeTemplate SEEDER e ekbar create hoy (NOT per product save):
     - Fashion category → 6 templates (size_chart, color_variants, fitting_guide, etc.)
     - Electronics category → 8 templates (warranty_period, product_condition, etc.)
     - UNIQUE constraint: (category_id, slug) — duplicate hobe na

  3. Product save korle SHUDHU ProductAttributeValue create hoy:
     - Each filled extra field = 1 row in product_attribute_values
     - References: attribute_template_id (FK) + value (text)

  4. Product edit/update korle:
     - Existing ProductAttributeValue DELETE for that product
     - NotUN values recreate koro (full replacement, same as current attribute system)

WHY THIS MODEL:
- "size_chart" template EKBAR toiri hoy, Fashion product er shomoy template create hoy NA
- Template create = seeder/setup time. Product save = data only.
- Duplicate prevention: UNIQUE(category_id, slug, is_global) on attribute_templates


### DECISION 1B: JSON ↔ attribute_templates SYNC MECHANISM (CRITICAL)
----------------------------------------------------------------------

PROBLEM:
  BusinessCategory.extra_fields JSON edit hole (admin panel theke nobe field add kore),
  AttributeTemplate table e sync na hole:
  - Form e noi field dekhabe ✅
  - Validation pass korbe ✅
  - Save korle matching template PABE NA → data silently hariye jabe ❌

SOLUTION: Idempotent Sync Job (Tenant-Aware)

  1. BusinessCategory UPDATED/CREATED event listener → SyncCategoryAttributeTemplates job dispatch
  2. Job runs SYNC for EACH TENANT (since attribute_templates is tenant-level):
     - Set tenant context → sync templates in that tenant's DB
     - extra_fields JSON theke template create koro (upsert)
     - JSON theke missing template → soft delete (is_active = false)
     - JSON e new field → create template (is_active = true)
  3. Idempotency: firstOrCreate + updateOrCreate pattern use koro

CODE PSEUDOCODE:

  class SyncCategoryAttributeTemplates implements ShouldQueue
  {
      public function handle(BusinessCategory $category): void
      {
          // CRITICAL: attribute_templates is TENANT table
          // Must sync for ALL tenants that use this category
          
          $tenants = Tenant::all();
          
          foreach ($tenants as $tenant) {
              $tenant->run(function () use ($category) {
                  $this->syncForTenant($category);
              });
          }
      }
      
      private function syncForTenant(BusinessCategory $category): void
      {
          $extraFields = $category->extra_fields ?? [];
          $fieldNames = collect($extraFields)->pluck('name')->toArray();

          // 1. UPSERT: extra_fields theke templates create/update
          foreach ($extraFields as $field) {
              AttributeTemplate::updateOrCreate(
                  [
                      'category_id' => $category->id,
                      'slug'        => Str::slug($field['name']),
                      'is_global'   => false,
                  ],
                  [
                      'name'              => $field['label'] ?? $field['name'],
                      'type'              => $field['type'],
                      'is_required'       => $field['required'] ?? false,
                      'placeholder'       => $field['placeholder'] ?? null,
                      'default'           => $field['default'] ?? null,
                      'options'           => $field['options'] ?? null,
                      'is_variant_option' => false,
                      'is_active'         => true,
                  ]
              );
          }

          // 2. SOFT DELETE: JSON theke removed fields → deactivate
          AttributeTemplate::where('category_id', $category->id)
              ->where('is_global', false)
              ->whereNotIn('slug', collect($fieldNames)->map(fn($n) => Str::slug($n)))
              ->update(['is_active' => false]);

          // 3. HARD DELETE: if template has no product_attribute_values → safe to remove
          AttributeTemplate::where('category_id', $category->id)
              ->where('is_global', false)
              ->where('is_active', false)
              ->whereDoesntHave('attributeValues')
              ->delete();
      }
  }

  // EventServiceProvider:
  // Handle BOTH created and updated events
  BusinessCategory::created(fn($cat) => SyncCategoryAttributeTemplates::dispatch($cat));
  BusinessCategory::updated(fn($cat) => SyncCategoryAttributeTemplates::dispatch($cat));

NOTE:
- Seeder runs ONCE (initial setup, creates templates in ALL tenant DBs)
- Sync job runs EVERY TIME admin edits extra_fields JSON (syncs to ALL tenants)
- Product save NEVER creates/edits templates — only reads existing ones
- product_attribute_values e data ache → template soft-delete (preserve data)
- product_attribute_values e data NAI → template hard-delete (cleanup)
- NEW category created → sync job creates templates for ALL existing tenants


### DECISION 2: stock_quantity Handling
-----------------------------------------

FINAL RULE:

  SINGLE PRODUCT (no variants):
    stock_quantity = REQUIRED, user directly fill korbe
    Rule: stock_quantity => required_if:has_variants,false|integer|min:0

  PRODUCT WITH VARIANTS:
    stock_quantity = AUTO-CALCULATED (sum of all variant stock_quantity)
    User ekhane stock fill korbe NA — variant table e fill korbe
    Controller auto-update korbe: $product->recalculateStock()

  DATABASE DEFAULT:
    stock_quantity DEFAULT 0

  EDGE CASE: Variant add/remove korle auto-recalculate hobe
  EDGE CASE: Variant stock change korle product-level stock auto-update hobe


### DECISION 3: SKU/Barcode Per-Tenant Uniqueness
---------------------------------------------------

ARCHITECTURE CONFIRMATION: Database-per-Tenant (stancl/tenancy v3.x)

  SocialBoost AI uses stancl/tenancy with DATABASE-PER-TENANT:
  - Each tenant has separate database (e.g., tenant1_socialboost, tenant2_socialboost)
  - products table exists in EACH tenant database separately
  - So unique:products,sku already works per-tenant automatically
  - No cross-tenant conflict possible

  VERIFIED:
  - config/tenancy.php → database config → separate DB per tenant
  - migrations run via tenants:migrate → per tenant DB

FINAL RULE:

  SKU:    Per-tenant unique (tenant DB theke automatic)
          Rule: sku => required|string|unique:products,sku
          No extra column needed — separate databases = separate unique constraints
          
  Barcode: Per-tenant unique (same logic)
           Rule: barcode => nullable|string|unique:products,barcode

  WHY: Tenant A "TSH-RED-S" use korte parbe, Tenant B o "TSH-RED-S" use korte parbe.
       Separate databases ache, so unique constraint already per-tenant work korbe.


### DECISION 4: Digital Product — Unit/Stock/Variant Handling
--------------------------------------------------------------

FINAL RULE:

  Category = "Digital Product / Service" hole:
    unit:       NOT REQUIRED (nullable, show koro NA)
    stock:      NOT REQUIRED (digital product e physical stock nai)
    variants:   OPTIONAL (user chaile "Basic Plan/Pro Plan" variant add korte parbe)
    variant.stock_quantity: NOT REQUIRED for Digital (default unlimited)

  IMPLEMENTATION:

  Controller e conditional logic:
  ```
  $isDigital = Category::find($request->category_id)?->slug === 'digital-product-service';

  // Unit: required unless Digital
  if (!$isDigital) {
      $rules['unit'] = 'required|string|max:50';
  }

  // Stock: required only if no variants AND not Digital
  $hasVariants = !empty($request->variants);
  if (!$hasVariants && !$isDigital) {
      $rules['stock_quantity'] = 'required|integer|min:0';
  }

  // Variant stock: required only if not Digital
  if ($hasVariants && !$isDigital) {
      $rules['variants.*.stock_quantity'] = 'required_with:variants|integer|min:0';
  } elseif ($hasVariants && $isDigital) {
      // Digital variant stock: optional, default 999999 (unlimited)
      $rules['variants.*.stock_quantity'] = 'nullable|integer|min:0';
  }
  ```

  VIEW: Unit dropdown + stock field → conditional hide/show based on category
        Alpine.js: category == "digital" → hide unit + stock fields
        Digital variant e stock field optional, default 999999

  STORAGE:
  - Digital product without variants: stock_quantity = 0 (or user input if provided)
  - Digital product with variants: each variant.stock_quantity = user input (default 999999)
  - Product.stock_quantity = SUM(variant.stock) if variants, else user input

## 5. FLOW: How It Works
===========================

### 5.1 Onboarding Flow
```
User Onboarding → Step 2: Business Info
  └── Select Business Category (Fashion / Electronics / Food / etc.)
        └── Category ID stored in BusinessSetting.category_id
        └── Category's extra_fields JSON stored in BusinessSetting.extra_fields_data
```


### 5.2 Product Create Flow
```
Product Controller
  └── Load user's BusinessSetting → get category_id (landlord DB integer)
  └── Load BusinessCategory from LANDLORD DB → get extra_fields JSON
  └── Load Category from TENANT DB by same ID → get category name/slug
  └── Determine category type (Digital? Physical? — check slug from tenant DB)
  └── Pass to view: $businessCategory, $category, $extraFields, $isDigital

Product Create View
  └── Layer 1: Render common fields
        └── Conditional: hide unit/stock if Digital
        └── Conditional: stock readonly if variants enabled
  └── Layer 1B: Render variant system (Color/Size/etc.)
        └── Default options: Color, Size (user can add more)
        └── Validate: combinations <= 100
  └── Layer 2: Render extra fields dynamically from $extraFields
        └── Alpine.js: category change hole extra fields update hobe

NOTE: Two category models exist:
  - BusinessCategory (landlord DB) — has extra_fields JSON, used for extra fields definition
  - Category (tenant DB) — has name/slug, used for product's category FK
  Both use SAME ID (BusinessCategory.id = Category.id) because seeder creates matching records.
```


### 5.3 Storage Flow (FINAL — Decision 1)
```
Form Submit
  │
  ├── Common fields → products table
  │
  ├── Variants → product_variants table + variant_attribute_values
  │     └── If variants exist: product.stock_quantity = SUM(variants.stock_quantity)
  │     └── If no variants: product.stock_quantity = user input
  │
  └── Extra fields → product_attribute_values table
        └── For each filled extra field:
              └── Find AttributeTemplate by (category_id, field_name)
              └── Create 1 ProductAttributeValue row:
                    - attribute_template_id: FK to template
                    - value: user input (text — see Note below)
        └── NO new AttributeTemplate created per product
        └── Templates were created ONCE in seeder
        └── If template not found → throw ValidationException (data protection)
```

NOTE on value column: product_attribute_values.value is TEXT type.
Boolean/number fields stored as string. For queries:
  - Boolean: whereRaw("value = '1'" or value IN ('true','1','on'))
  - Number: CAST(value AS DECIMAL) for comparisons


## 6. STORAGE MODEL — COMPLETE
================================

### 6.1 AttributeTemplate (Template — Created ONCE + Synced)

```sql
attribute_templates:
├── id (bigint, PK)
├── category_id (FK → categories.id, NULL for global)
├── is_global (boolean, default: false)
├── is_variant_option (boolean, default: false)
├── name (string)
├── slug (string)
├── type (enum: text, number, select, boolean, date)
├── options (JSON, nullable) -- for select type
├── is_required (boolean, default: false)
├── is_active (boolean, default: true) ← for soft-delete on sync
├── sort_order (integer)
├── created_at, updated_at
│
├── UNIQUE: (category_id, slug, is_global)
│   NOTE: NULL category_id = global options
│   MySQL/Postgres: NULL != NULL in unique constraints
│   SOLUTION: Use firstOrCreate + application-level guard (see Decision 1B)
│
└── Relationships:
    ├── category() → Category (nullable)
    └── attributeValues() → ProductAttributeValue (for checking data before hard-delete)
```


### 6.2 ProductAttributeValue (Data — Created PER PRODUCT)

```sql
product_attribute_values:
├── id (bigint, PK)
├── product_id (FK → products.id, CASCADE delete)
├── attribute_template_id (FK → attribute_templates.id, CASCADE delete)
├── value (text) ← all types stored as text (boolean→'1'/'0', number→'5')
├── created_at, updated_at
│
└── UNIQUE: (product_id, attribute_template_id)
```

NOTE on value storage:
  - text: stored as-is ("Hello World")
  - number: stored as string ("5", "3.14") → CAST for queries
  - boolean: stored as '1'/'0' (or 'true'/'false') → whereRaw for queries
  - select: stored as selected option string ("Cotton")
  - date: stored as date string ("2025-01-15")

Query examples:
  - Filter by boolean: ->where('value', '1')
  - Filter by number: ->whereRaw('CAST(value AS DECIMAL) > ?', [7])
  - Filter by text: ->where('value', 'LIKE', '%keyword%')

```php
// UPDATE rules — exclude current product from unique check
$rules['sku'] = 'required|string|unique:products,sku,' . $product->id;
$rules['barcode'] = 'nullable|string|unique:products,barcode,' . $product->id;
```


### 8.2 Unit & Stock (Conditional — Decision 2 & 4)
```php
$isDigital = Category::find($request->category_id)?->slug === 'digital-product-service';
$hasVariants = !empty($request->variants);

// Unit: required unless Digital
if (!$isDigital) {
    $rules['unit'] = 'required|string|max:50';
}

// CRITICAL: Mutually exclusive stock rules
// If variants exist → stock auto-calculated, DO NOT validate stock_quantity at all
// If no variants → stock required for physical, optional for digital
if (!$hasVariants) {
    if (!$isDigital) {
        // Single physical product → stock REQUIRED
        $rules['stock_quantity'] = 'required|integer|min:0';
    } else {
        // Single digital product → stock OPTIONAL
        $rules['stock_quantity'] = 'nullable|integer|min:0';
    }
}
// NOTE: If $hasVariants is true, stock_quantity is NOT added to rules at all
// This prevents validation conflict where both required + nullable could fire
```


### 8.3 Variant Fields
```php
$rules['variants']                     = 'nullable|array';
$rules['variants.*.sku']               = 'required_with:variants|string|max:255|unique:product_variants,sku';
$rules['variants.*.price']             = 'nullable|numeric|min:0';
$rules['variants.*.attributes']        = 'nullable|array';
$rules['variants.*.barcode']           = 'nullable|string';

// Digital category → stock optional (default 999999)
if ($isDigital) {
    $rules['variants.*.stock_quantity'] = 'nullable|integer|min:0';
} else {
    $rules['variants.*.stock_quantity'] = 'required_with:variants|integer|min:0';
}
```

### 8.5 Category-Specific Extra Fields (Dynamic)
```php
// Load extra fields from BusinessCategory
$extraFields = $businessCategory->extra_fields ?? [];
$extraRules = [];

foreach ($extraFields as $field) {
    $rule = 'nullable';  // default
    if ($field['required'] ?? false) {
        $rule = 'required';
    }
    
    switch ($field['type']) {
        case 'text':
        case 'textarea':
            $rule .= '|string|max:1000';
            break;
        case 'number':
            $rule .= '|numeric|min:0';
            break;
        case 'boolean':
            $rule .= '|boolean';
            break;
        case 'select':
            if (!empty($field['options'])) {
                $rule .= '|in:' . implode(',', $field['options']);
            }
            break;
    }
    
    $extraRules["extra.{$field['name']}"] = $rule;
}

$rules = array_merge($rules, $extraRules);
```

### 12.2 No Category Selected
- Extra fields section hidden
- Variant system still works
- All common fields visible

### 12.3 Category Change Mid-Form
- Extra fields section auto-update (Alpine.js)
- Already filled extra fields → warning: "Category changed, previous extra fields will be lost"

### 12.4 Single Product (No Variants)
- User chaile variant skip korte parbe
- Single product form e stock_quantity directly fill korbe (REQUIRED for physical)
- stock_quantity => required_if:has_variants,false

### 12.5 Digital Product + Variants
- Digital products e usually variant lage na
- But user chaile (e.g., "Basic Plan/Pro Plan") variant add korte parbe
- unit/stock fields hidden for Digital category
- Variant stock optional (default 999999 unlimited)

### 12.6 discount_price > base_price
- Frontend: real-time validation, red border + error
- Backend: validation error "Discount price must be less than or equal to base price"

### 12.7 50+ Variant Combinations
- Frontend soft warning: "50+ variants generate hocche. Are you sure?"
- User can continue OR reduce options
- 100+ → Hard block, cannot submit

### 12.8 Weight for Shipping
- Common field in Layer 1
- All categories can use for shipping cost calculation
- Furniture-er weight_kg extra field REMOVED (duplicate)

### 12.9 Value Type Casting
- product_attribute_values.value is TEXT type
- Boolean fields: where('value', '1') for true
- Number fields: whereRaw('CAST(value AS DECIMAL) > ?', [7]) for comparisons
- Future: may need a typed_value accessor or computed column

### 12.10 Product Listing — Extra Fields Not Shown (Intentional)
- Product index page shows: name, SKU, price, stock, status, brand, category
- Extra fields NOT shown in listing (too many columns, performance concern)
- Extra fields visible in: product show page, product edit form
- Future: optional column toggle in listing (admin preference)
